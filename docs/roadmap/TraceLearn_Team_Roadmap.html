01 · Overview

What we are building

TraceLearn is a learning Agent for a university production internship. It is not a commercial launch — it is optimised for a working demo, explainability, technical correctness, defence quality, and portfolio value. The product is not "AI makes a study plan" (already common). The product is a material-grounded learning Agent that creates, tracks, and explains the evolution of a learner's path.

The question we answer

"Why did my learning plan change?" — not only "what should I study?"

The differentiator

Evidence-driven, material-grounded, explainable learning path evolution.

The team

Three members, one short internship, one small complete system.

02 · Problem

Why existing tools fall short

Generic AI planners

A single prompt produces a study plan. It ignores the learner's actual course materials and never adapts to what the learner actually does. "AI makes a plan" is a commodity.

No accountability

When a plan changes, learners get a new plan with no reason. There is no link between evidence, the decision, and the change. Trust and learning both suffer.

TraceLearn closes the loop that others leave open: plan → study → evidence → visible, justified adaptation.

03 · Value Proposition

The trace is the product
Evidence→ Agent reasoning→ New plan version→ Explanation→ Version history

Material-grounded

Plans, questions and tasks are built from the learner's own syllabus concepts.

Evidence-driven

Real study behaviour — completed tasks, quiz results, time — drives replanning.

Explainable

Every plan change is traced to specific concepts and specific evidence.

Supporting capability

Cross-language material grounding

Study material in one language, explanations in another — with technical terms preserved. The concept map is language-neutral: the canonical term is the grounding anchor, while explanations, tasks and the Agent's reasoning adapt to the learner's language. Two languages (Chinese, English), content-only. This sharpens the material-grounded idea; it is not the product identity.

04 · User Journey

Onboarding (Option C hybrid)

Rule: never show a roadmap before concept extraction, never make the user wait on a blocking spinner. Aha moment must land within about three minutes.

Step 1 · ~10s

Goal setup

Enter goal, deadline, weekly hours. Optionally drop one PDF/TXT — processing starts in the background.

Step 2 · while processing

Warm-up questions

2–3 lightweight self-reflection questions. Context only — not the diagnostic, not plan input.

Step 3

Concept map appears

The AI extracts a concept map from the material (or from the goal topic if no material).

Step 4 · human-in-the-loop

Confirm / edit concepts

The learner reviews and edits the concept map. The confirmed map is authoritative.

Step 5

Diagnostic

5–8 concept-targeted questions produce a per-concept ability signal (heuristic, not a measurement).

Step 6 · aha moment

Roadmap Version 1

A focused roadmap with concept-tagged daily tasks and flagged weak areas.

05 · Concept Map

The spine that connects everything

The concept map extracted from the material is a first-class database entity. It is the shared key that links every part of the system. This is what makes replanning pedagogically meaningful instead of mere schedule juggling.

Material→ Concepts→ Diagnostic questions→ Learning tasks→ Evidence→ Agent decisions

Worked example — the whole product in one row

ConceptDiagnosticTaskEvidenceAgent decision
Database normalization Student fails the normalization question Add normalization practice Quiz score stays low Create extra normalization remediation tasks — and explain why

With material: targeted diagnosis, concept-linked tasks, focused roadmap, meaningful replanning. Without material: the system falls back to goal-topic concepts and still works end-to-end.

Language-neutral anchor

  • canonical_term — the technical term preserved verbatim (e.g. "Normalization")
  • the join key across diagnostics, tasks, evidence, decisions
  • used in the Agent's reasoning and tool arguments — always English

Adapts to the learner

  • explanation — written in the learner's language
  • task descriptions, diagnostic questions, reasoning shown to the user
  • Chinese or English; technical terms stay in their original form

07 · The Agent

Why this is a real Agent, not a chatbot

A wrapper is one prompt in, one answer out. TraceLearn's Agent reads external state through tools, decides whether a plan needs to change, and acts on persistent state through a triggered loop. We use deterministic orchestration: code calls the read tools in a fixed order and the LLM owns a single decision point (change / no-change, the reasoning, and the proposed plan). This is genuinely tool-using and agentic, without the fragility of free-form tool looping — a stronger, more defensible story than unbounded autonomy.

Evidence→ Deterministic trigger→ Hermes reasoning→ Validator→ New version + decision

The control split

Deterministic code controls

  • When the Agent is invoked (trigger thresholds)
  • Validation of any proposed plan (dates, hours, concepts)
  • Persistence, versioning, parent linkage
  • Guardrails — reject impossible schedules

The LLM decides

  • Whether the plan needs to change
  • What specifically to change
  • The natural-language justification
  • Interpretation of ambiguous evidence

One-line defence: we never let the LLM run unbounded — deterministic triggers decide when it wakes up, and deterministic validators check everything it proposes before it is persisted.

10 · Execution Plan

48-hour MVP sprint (immediate execution)

Phase 0 (the code seed) is already delivered. What follows is the approved 48-hour build. Hours are guides, not gates. Build the loop from both ends toward the middle: validator + triggers early (safe, testable), the live LLM decision point last (behind a mock until integration).

Hours 0–2 · all

Freeze contracts, stand up stubs

Freeze the 3 JSON shapes (PlanVersionOut, AgentDecisionOut+trace, ConceptOut). Run the seed backend; all endpoints return mock data. After this, everyone works in parallel.

Hours 2–8 · parallel

Build against mocks

A validator + triggers (unit-tested) + orchestrator skeleton B screens against mocks C extraction + the 3 generation prompts vs MOCK_LLM.

Hours 8–20 · integrate

Real LLM, one seam at a time

Concepts → diagnostic → roadmap wired to real Hermes. A wires real tools + append-only versioning.

Hours 20–32 · the loop

Close the agent loop end-to-end

Simulate → trigger → decision → V2 → record. B builds the diff + tool-trace viewer against the real decision JSON.

Hours 32–42 · harden

Seed, fallback, bilingual field

Harden seed + simulate; build the recorded/seeded fallback demo; wire the explanation-language field.

Hours 42–48 · freeze

Rehearse twice, fix demo-path only

Freeze scope. Rehearse the one normalization-remediation scenario twice. Fix only demo-path bugs.

Failure plan (if behind schedule)

−25% time

Cut live bilingual demo, live upload processing, dashboard charts; use seeded concepts + fixed diagnostic. Keep confirm → V1 → simulate → V2 → diff → trace.

−50% time

Seed everything before the agent. Only live LLM call = the decision (change/no-change + reasoning for V2). Demo starts at "here is V1".

Irreducible floor

Seeded V1 + simulate button + one live agent decision → V2 + reasoning + diff + trace. If that runs, the differentiator is proven.

11 · Demo Scenario

The five minutes that decide the grade
Show the grounded plan. A roadmap built from the student's own slides — Version 1, with concept-tagged tasks.
Inject evidence live. Use the simulate control: fail a normalization quiz, mark normalization tasks incomplete, advance the clock.
The trigger fires. Deterministic code detects the learner is behind / weak on a concept and wakes the Agent.
Show the tool trace. The Agent reads state, reasons over the concept, and proposes a change — every tool call is visible in order.
Version 2 is created. Side-by-side diff vs Version 1: normalization remediation tasks added.
Answer the question. The written explanation says exactly why the plan changed, cited to concept and evidence.

A recorded / seeded fallback path exists so the demo never depends on a live model call at the worst moment.

Bilingual variant

Same flow, one field changed: upload an English database PDF with explanation language set to Chinese. The concept map keeps English canonical terms while explanations, tasks and the Agent's reasoning appear in Chinese — showing cross-language grounding with no extra demo machinery.
